\ssc{總論}
\sssc{Resource Acquisition Is Initialization (RAII)}
RAII is a programming idiom in C++ used to manage resource lifetimes automatically. The key idea is:
\begin{itemize}
    \item Resources (e.g., memory, file handles, sockets) are acquired in the constructor of a class.
    \item Resources are released in the destructor, ensuring deterministic cleanup and exception safety.
\end{itemize}
RAII is applied throughout my STL implementation in all classes and structures to manage resources safely and efficiently.
\sssc{Substitution Failure Is Not An Error (SFINAE)}
SFINAE is a compile-time technique in template programming. If substituting template parameters results in an invalid type or expression, the compiler silently ignores that overload instead of producing an error. This mechanism enables selective template instantiation and overload resolution.
\sssc{Concepts}
C++20 introduces \textbf{concepts}, which allow explicit constraints on template parameters in a readable and expressive way. Concepts replace some uses of SFINAE with clearer syntax. Key benefits include:
\begin{itemize}
    \item Clear compile-time diagnostics.
    \item Explicit template constraints.
    \item Improved code readability and maintainability.
\end{itemize}
In my STL implementation, concepts are heavily used to determine the appropriate overload or \texttt{if constexpr} branch. This technique enables optimizations for trivially copyable types, contiguous iterators, allocator traits-based optimization, etc.
\sssc{Rule of Three / Five / Zero}
\begin{itemize}
    \item \textbf{Rule of Three:} If a class defines a destructor, copy constructor, or copy assignment operator, it likely needs all three.
    \item \textbf{Rule of Five:} Extends the Rule of Three by including move constructor and move assignment for C++11 and later.
    \item \textbf{Rule of Zero:} Prefer using RAII and standard containers to avoid manually managing special member functions.
\end{itemize}
These rules are applied consistently in all containers of my STL implementation to ensure correct and safe resource management.
\sssc{Class Template Argument Deduction (CTAD)}
C++17 introduced \textbf{Class Template Argument Deduction (CTAD)}, allowing the compiler to deduce template parameters from constructor arguments. STL containers provide \textbf{deduction guides} to assist this process, which are also implemented in my library to improve usability.
\sssc{Exception Safety Guarantees and \texttt{noexcept}}
STL containers follow strict \textbf{exception safety guarantees} during operations such as insertion, deletion, or reallocation:
\begin{itemize}
    \item \textbf{No-throw guarantee:} The operation never throws an exception.
    \item \textbf{Strong guarantee:} If an exception occurs, the container's state remains unchanged.
    \item \textbf{Basic guarantee:} Some changes may occur, but the container remains in a valid state.
\end{itemize}
In my implementation, I use \texttt{try-catch} blocks and proper deallocation to handle exceptions, prefer \texttt{noexcept} branches when possible, and provide explicit \texttt{noexcept} specifications.
\sssc{\texttt{const}, \texttt{static}, \texttt{constexpr}, \texttt{inline}, and \texttt{explicit}}
These keywords are applied appropriately throughout my code to enforce immutability, enable compile-time evaluation, reduce code duplication, prevent unintended implicit conversions, etc.
\sssc{Lvalue and Rvalue References}
Lvalue and rvalue references are used extensively in function parameters and member functions to enable move semantics, perfect forwarding, and efficient resource management.
\sssc{Argument-Dependent Lookup (ADL)}
ADL, also known as \textbf{Koenig lookup}, allows the compiler to search for functions not only in the current scope but also in the namespaces of the function argument types. This enables intuitive and unqualified function calls.
\sssc{Structured Bindings}
Structured bindings, introduced in C++17, allow unpacking of tuples, pairs, arrays, or user-defined types into individual variables. In my STL implementation, \verb|include/tuple-using.hpp| provides \verb|std::tuple_size| and \verb|std::tuple_element| support for \verb|mystd::array| and \verb|mystd::tuple|.
\sssc{\texttt{auto}}
The \verb|auto| keyword allows the compiler to deduce variable types from initializers. C++14 introduced \verb|auto| return type deduction, but all return statements must produce the same type according to the standard.  
\sssc{\texttt{decltype(auto)}}
\verb|decltype(auto)| preserves the exact type of the initializer, including references, and is used together with perfect forwarding and template overloads in \verb|mystd::tuple| and other components.
\sssc{Empty Base Optimization (EBO)}
In C++, an empty class normally has size 1 to ensure unique addresses. EBO is a compiler optimization that allows empty base classes to occupy no additional storage when used as a base. In my implementation, EBO is applied in \verb|_tuple_impl|, the internal recursive inheritance mechanism for \verb|mystd::tuple|.
\sssc{\texttt{[[no\_unique\_address]]}}
Introduced in C++20, this attribute allows empty member objects to be optimized similarly to empty bases, reducing storage overhead. It is used for allocator members in my STL containers.
\sssc{Template Compile-Time Recursion}
Templates can perform computation at compile time using recursion:
\begin{itemize}
    \item Each template instantiation computes one step of the recursion.
    \item Recursion terminates at a base case specialization.
    \item Computation occurs entirely at compile time, producing constant expressions.
\end{itemize}
This technique is used in \verb|include/tuple.hpp| for \verb|_tuple_impl|, \verb|tuple|, and related non-member functions.







\ssc{include/algorithm-copy.hpp}
The \verb|algorithm-copy.hpp| file provides three basic data transfer algorithms: \verb|copy|, \verb|copy_if|, and \verb|copy_backward|, mimicking the interface design of C++20 \verb|<algorithm>|. The copy function family demonstrates the polymorphism programming techniques, particularly conceptual parameter constraints and conditional optimization strategies.

I utilize concepts for constraints, such as \verb|std::input_iterator|, \verb|std::weakly_incrementable|, \verb|std::bidirectional_iterator|, and \verb|std::indirectly_writable| in the definition, and
\verb|std::contiguous_iterator| and \verb|std::is_trivially_copyable_v| for specialized acceleration.

In \verb|copy| algorithms, I use \verb|std::memcpy| when possible and element-wise assignment only when necessary, as demonstrated below (excerpted from \verb|include/algorithm-copy.hpp| lines 7-32)
\begin{lstlisting}[language=C++]
template <std::input_iterator InputIt, std::weakly_incrementable OutputIt>
  requires std::indirectly_writable<OutputIt, std::iter_value_t<InputIt>>
constexpr OutputIt copy(InputIt first, InputIt last, OutputIt d_first) {
  if constexpr (std::contiguous_iterator<InputIt> &&
                std::contiguous_iterator<OutputIt> &&
                std::is_trivially_copyable_v<
                    typename std::iterator_traits<InputIt>::value_type> &&
                std::is_same_v<
                    typename std::iterator_traits<InputIt>::value_type,
                    typename std::iterator_traits<OutputIt>::value_type>) {
    if (first == last)
      return d_first;
    const auto *src = std::to_address(first);
    const auto *src_end = std::to_address(last);
    auto *dest = std::to_address(d_first);
    const std::size_t count = (src_end - src) * sizeof(*src);
    std::memcpy(dest, src, count);
    return d_first + (last - first);
  }
  while (first != last) {
    *d_first = *first;
    ++first;
    ++d_first;
  }
  return d_first;
}
\end{lstlisting}

Similarly, in \verb|copy_backward| algorithm, \verb|std::memmove| is used when possible, as demonstrated below (excerpted from \verb|include/algorithm-copy.hpp| lines 49-73)
\begin{lstlisting}[language=C++]
template <std::bidirectional_iterator BidirIt1,
          std::bidirectional_iterator BidirIt2>
constexpr BidirIt2 copy_backward(BidirIt1 first, BidirIt1 last,
                                 BidirIt2 d_last) {
  using ValueType1 = typename std::iterator_traits<BidirIt1>::value_type;
  using ValueType2 = typename std::iterator_traits<BidirIt2>::value_type;
  if constexpr (std::contiguous_iterator<BidirIt1> &&
                std::contiguous_iterator<BidirIt2> &&
                std::is_trivially_copyable_v<ValueType1> &&
                std::is_same_v<ValueType1, ValueType2>) {
    auto n = last - first;
    if (n > 0) {
      std::memmove(std::addressof(*d_last) - n, std::addressof(*first),
                   n * sizeof(ValueType1));
    }
    return d_last - n;
  } else {
    while (first != last) {
      --last;
      --d_last;
      *d_last = *last;
    }
    return d_last;
  }
}
\end{lstlisting}
